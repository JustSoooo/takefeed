# 监控 & 选股平台 · Claude Code 开发指南（Guidebook）

> 本文档是交给 Claude Code 的完整开发说明书。覆盖 V1-V4 四个模块的功能定义、数据源、
> 技术架构、前端设计规范（基于 design-taste-frontend skill 适配数据面板场景）、
> 以及开发顺序与验收标准。按模块顺序渐进开发，每个版本独立可用。

---

## 0. 项目定位与总原则

**这是一个决策支持系统，不是自动交易系统。** 它回答三个递进的问题：

1. 现在该不该积极介入市场？（V1 宏观情绪）
2. 该介入的话，钱往哪个行业/板块去？（V2 行业轮动）
3. 选定方向后，具体买哪只票、什么时候要警惕？（V3 个股评分 + V4 股池监控）

**背景**：使用者是一名有多年实盘经验的个人投资者，实盘验证过"收益主要来自选股而非择时"
（同一套技术信号在 MU/SNDK 上大赚、在 COHR/LITE/KORU 上亏损）。此平台的价值是把
他手动做的信息收集/判断流程系统化，不是替他做决定。

**总原则（全模块适用）：**

- **数据与判断分层**：取数、计算、打分全部用确定性代码；LLM（Claude API）只负责
  把结构化结果翻译成可读的中文叙事和建议基调，绝不让 LLM 参与数字计算
- **所有历史落地**：每天的原始指标值全部存 SQLite，为后续验证"这套框架准不准"留证据
- **权重阈值全部进配置文件**（config.yaml），不硬编码；初期不定死权重，跑几周人工
  校准后再固化
- **覆盖市场**：V1-V2 先美股；A股（akshare 数据源）作为各模块的二期扩展，
  代码结构上预留 market 抽象层（fetcher 接口统一，实现按市场分开）
- **不做的事**：不接交易接口、不自动下单、不做实时 tick 级数据（日级/小时级足够）

## 1. 模块总览与开发顺序

| 模块 | 名称 | 回答的问题 | 开发优先级 |
|---|---|---|---|
| V1 | 宏观/市场情绪仪表盘 | 该不该介入、该多积极 | P0，先做 |

## 2. V1 · 宏观/市场情绪仪表盘

### 2.1 核心产出

一个 0-100 的"风险偏好综合评分" + 六维度分解 + 一段 LLM 生成的中文判断与操作建议。
输出两种形态：每日 markdown 报告存档 + 静态 HTML 仪表盘。

### 2.2 六维度指标与数据源

| 维度 | 指标 | 数据源 | 频率 |
|---|---|---|---|
| 波动率 | VIX 绝对值 + 近1年百分位；VIX9D/VIX 比值 | yfinance `^VIX` `^VIX9D` | 每日 |
| 趋势 | SPY/QQQ 相对 20/50/200 日均线位置、均线排列 | yfinance | 每日 |
| 广度 | 标普500成分股站上50日/200日均线的比例 | yfinance 批量拉成分股自算 | 每日 |
| 信用 | HYG/IEF 比价及其 5/20 日变化方向 | yfinance | 每日 |
| 板块轮动 | 11个 SPDR 板块 ETF 近1周/1月涨跌排序 | yfinance | 每日 |
| 情绪 | AAII 看多/看跌比；CNN Fear & Greed | 网页抓取（注意频率） | 周/日 |

**广度指标的实现提示**：标普500全量成分股第一次拉取量大，先用道指30或纳指100
练手跑通逻辑，再扩全量。成分股清单可从 Wikipedia 的 S&P 500 constituents 页面抓取，
落地本地后定期（月度）刷新即可，不用每天抓清单。

### 2.3 评分逻辑

- 每个维度输出：原始值 → 历史百分位（近1年）→ 维度分（-20 到 +20 区间）
- 综合分 = 50 + Σ(维度分 × 权重)，clamp 到 0-100
- 三档状态：<35 防守 / 35-65 谨慎观望 / >65 积极
- **权重初始全部等权**，config.yaml 里可调，跑2-4周人工复盘后再调整

### 2.4 叙事生成（Claude API）

输入：六维度的结构化打分结果（JSON）。
输出：150-250字中文判断段落 + 2-3条可执行建议（仓位基调、触发条件式的预案）。
Prompt 要求输出里引用具体维度数据支撑结论，禁止输出模糊套话。

## 3. V2 · 行业/板块轮动追踪

### 3.1 核心产出

回答"过去N周资金持续流向哪些板块、这个趋势还在不在"。

- **板块相对强弱矩阵**：11个 SPDR 板块 ETF 相对 SPY 的超额收益，按 1周/1月/3月
  三个窗口各排一列，看排名的持续性（连续N周排名前3 = 强持续信号）
- **板块内部健康度**：强势板块内部是普涨还是靠一两只权重股拉动
  （板块内成分股上涨家数占比）
- **RRG 风格的轮动象限图**（可选进阶）：相对强度 vs 相对强度动量的散点，
  四象限：领涨/走弱/落后/改善

### 3.2 数据源

- 板块 ETF 行情：yfinance
- 板块成分股清单：SPDR 官网 holdings CSV（各板块 ETF 官方页面提供下载）
- A股二期：申万一级行业指数，akshare `sw_index_daily`

### 3.3 与 V1 的联动

V1 的"板块轮动"维度只看当前快照；V2 把它做成时间序列 + 持续性判断。
V2 完成后，V1 的板块维度打分可直接引用 V2 的持续性结论（数据层共享 SQLite）。

## 4. V3 · 个股综合评分（基本面 + 舆情）

### 4.1 核心产出

针对候选池标的（初期 = 用户手工维护的 watchlist，V2 成熟后可从强势板块自动生成候选），
输出每只票的结构化评分卡：

| 评分块 | 内容 | 数据源 |
|---|---|---|
| 财务动量 | 营收/EPS 同比与环比趋势、最近一期是否 beat | yfinance `earnings_history` `quarterly_income_stmt` |
| 机构态度 | 分析师目标价中位数 vs 现价、近30日评级上调/下调 | yfinance `analyst_price_targets` `upgrades_downgrades` |
| 相对强度 | 个股 vs 所属板块 ETF 的 1月/3月超额收益 | yfinance |
| 舆情摘要 | 近7日新闻标题聚合 → LLM 归纳利多/利空/中性 | yfinance `ticker.news` + web search 补充 |
| 事件日历 | 下次财报日期、距今天数 | yfinance `calendar` |

### 4.2 舆情模块的实现纪律

- LLM 只做"归纳分类已抓到的新闻"，不做"预测涨跌"
- 每条归纳必须附原始新闻标题作为溯源，禁止 LLM 输出无来源的"市场传闻"
- 输出三档标签（利多/利空/中性）+ 一句话理由，不输出情绪分数
  （伪精确的数字情绪分会诱导过度信任）

## 5. V4 · 目标股池监控 + 异常预警

### 5.1 核心产出

对 watchlist（用户手工维护，或从 IBKR/Futu 持仓导出 CSV 导入）做每日巡检，
触发规则时生成告警条目（进每日报告 + 可选推送）。

### 5.2 预警规则（初始集，config 可增删）

- 价格突破/跌破 20日、50日均线（收盘确认，非盘中）
- 单日成交量 > 30日均量 2 倍
- 单日涨跌幅超过该股 90 日波动率的 2 个标准差
- 距离财报日 ≤ 5 个交易日（提示"财报临近，注意仓位"）
- 分析师评级变动（升/降级发生当日）
- 舆情异动：单日新闻条数 > 近30日日均 3 倍（复用 V3 舆情抓取）

### 5.3 推送通道（可选，二期）

优先本地方案：每日报告生成后通过邮件（SMTP）或 Telegram Bot 发送摘要。
不做实时盘中推送（日级巡检定位，避免做成盯盘工具）。

## 5.5 · 研报/分析资源摄取层（横向管道，服务 V1/V2/V3）

### 定位

使用者拥有市场分析资源（券商研报、机构分析报告、付费专栏等 PDF/网页/文章形态）。
本层不是第五个独立模块，而是一条横向数据管道：把非结构化的分析内容结构化后，
分发给 V1（宏观观点）、V2（行业观点）、V3（个股卖方逻辑）三个模块作为**参考信息层**展示。

### 管道设计

```
输入(PDF / 网页URL / 文章文本 / 截图)
  → 解析(pdfplumber 提取文本；截图走 OCR；网页走 readability 抽正文)
  → LLM 结构化摘要(输出固定 schema JSON)：
      { 来源, 发布日期, 覆盖层级(宏观/行业/个股), 覆盖对象(板块名或ticker),
        核心观点[1-3条], 关键论据[每条观点附], 方向(看多/看空/中性),
        事实类陈述[] , 判断类陈述[] }
  → 打标入库(SQLite research_notes 表，宏观/行业/ticker 三级标签索引)
  → 分发展示：
      V1 报告尾部"近期机构宏观观点"区块
      V2 板块页"研报观点 vs 量化信号一致性"对照区块
      V3 个股评分卡新增"卖方逻辑"字段(近30日该股观点聚合)
```

输入入口：`ingest_research.py` 命令行工具，支持单文件/目录批量/URL 三种方式，
使用者拿到新报告后手动跑一次即可入库（不做自动爬取，版权和稳定性都是坑）。

### 硬性纪律（本层比量化数据更容易失真，以下规则不可妥协）

1. **观点与事实分离存储**：LLM 摘要时强制区分"报告陈述的事实"(财报数字、行业数据)
   与"报告的判断"(目标价、评级、预测)，schema 中分字段存放，页面展示时视觉区分
2. **强制溯源**：每条观点展示时必须带 来源名+发布日期，可展开查看原文摘录段落；
   LLM 归纳无法定位到原文出处的观点直接丢弃不入库
3. **时效衰减**：观点按发布日期计算年龄，>30 天降权展示(灰度)，>60 天默认折叠；
   研报观点半衰期短，过期逻辑比没有逻辑更危险
4. **观点不进评分公式**：研报聚合结果仅作为参考信息层展示，永远不进入
   V1 综合分或 V3 评分卡的数字计算——卖方研报存在系统性看多偏差(结构性问题)，
   混入量化打分会污染整个体系的客观性
5. **一致性对照而非替代**：V2 的对照区块回答的问题是"研报观点和量化信号
   是否指向一致"，一致时增强信心，冲突时提示人工深入研究，系统不裁决谁对

### 5.4 期权信息穿插（依赖 V5 数据层，V5 完成后回填）

- **V3 个股评分卡新增"期权市场情绪"字段**：Put/Call 成交量比、IV Rank
  (当前隐含波动率在近1年区间的百分位)、期权总持仓量变化——期权市场常领先反映
  机构对个股的方向预期
- **V4 预警规则新增两条**：IV 单日跳升 > 20%（市场在为该股定价某个事件）；
  单日期权成交量 > 30日均量 3 倍且集中于单一行权价（异常大单活动）

## 5.6 · V5 期权分析模块

### 定位

针对已选定的标的（通常来自 V3 评分卡筛出的候选，或 watchlist 内标的），回答三个问题：
这只票有哪些**值得交易**的期权合约？我对未来某个日期的股价有预期时，
不同合约/组合的**预估盈亏**是多少？按我的期限和风险偏好，**推荐哪些腿策略**？

### 数据源（抽象接口，双实现）

```
core/fetchers/options_base.py     # OptionsFetcher 抽象接口
├── options_financial_service.py  # 首选：financial-service（实现时接入，
│                                 #   数据质量和希腊值精度优于免费源）
└── options_yfinance.py           # 降级：yfinance ticker.option_chain()
                                  #   (免费；IV和希腊值质量一般，价差数据可用)
```

无风险利率：取 ^IRX(13周国库券) 或 financial-service 提供的利率曲线。

### 功能一：期权链浏览

- 按到期日分组展示：行权价、买/卖价、中间价、成交量、持仓量(OI)、IV、Delta
- **流动性过滤（默认开启，硬规则）**：OI < 100 或 当日成交量 < 10 或
  买卖价差 > 中间价 10% 的合约默认折叠隐藏——无流动性合约的理论盈亏毫无实盘意义，
  进出场滑点会吞掉全部纸面利润
- IV 曲面速览：各到期日的 ATM IV 连线（期限结构），判断近月是否因事件（财报）被抬高

### 功能二：预期情景收益预估

输入：标的、预期股价（或区间）、预期到达日期、可选的持仓组合（1-4 腿）。
输出：

- **到期 payoff 图**（复用本地 options-payoff skill 的渲染逻辑）
- **预期日期的中途估值**：用 Black-Scholes 计算预期日那天（未到期时点）
  各腿的理论价值，汇总组合盈亏——这是和"到期payoff"的本质区别，
  多数散户工具只画到期图，忽略了中途平仓才是常态
- **双 IV 情景强制展示**：① IV 维持当前水平 ② IV 回落至近1年中位数。
  两个情景的盈亏并排展示。若预期日期横跨财报日，页面顶部强制显示
  "跨财报持仓，IV crush 风险" 警示条
- 敏感性小表：预期价格 ±5%、预期日期 ±5 个交易日的盈亏矩阵

### 功能三：腿策略推荐（3×3 矩阵）

按 **到期分桶 × 风险偏好** 输出候选策略矩阵，每格 1-2 个具体组合：

| | 保守 | 中性 | 激进 |
|---|---|---|---|
| **末日 (0-3 DTE)** | 不推荐（该格永远输出"不建议"+理由） | 深度价内单腿/宽跨式卖方对冲版 | 价外单腿、末日垂直价差 |
| **中期 (2周-3月)** | 备兑开仓、现金担保卖put | 垂直价差(bull call/bear put) | 单腿买方、比例价差 |
| **长期 (6月+/LEAPS)** | 深度价内LEAPS替代持股 | LEAPS+短期卖call(PMCC) | 长期价外单腿 |

推荐生成规则：

- 每个推荐必须输出：具体腿（行权价+到期日+方向）、**最大亏损**、最大盈利、
  盈亏平衡点、当前中间价成本、所需保证金估算、选择理由（引用 IV rank、
  期限结构、V3 评分卡结论等具体数据）
- **最大亏损用最显著的视觉层级展示**，先于最大盈利
- 末日档整体带高风险提示；保守×末日格子硬编码为"不建议"——
  末日期权与保守偏好本质矛盾，系统不提供伪保守方案
- 流动性过滤同样适用：推荐的每条腿必须通过功能一的流动性阈值
- 推荐附带与 V3 评分卡方向的一致性标记（评分卡看多但用户要求看空策略时，
  照常输出但标注分歧）

### 硬性纪律与模型局限声明（页面固定展示，不可折叠）

1. 所有盈亏为**理论模型估算**：BS 模型不处理美式提前行权、股息除权、
   盘后跳空；实际成交价受价差和滑点影响
2. 预估收益强依赖 IV 假设，IV 情景切换可导致结论反转
3. 推荐矩阵是**候选清单**而非投资建议，最终决策和风险自担；
   系统不追踪用户实际持仓、不提供追加保证金测算
4. 卖方策略（卖put/卖call）的保证金和被指派风险需在券商端自行确认，
   本系统的保证金估算仅为量级参考

```
│   │   └── summarize.py        # LLM 结构化摘要(固定schema)
│   └── narrative/
│       └── generate.py         # Claude API 叙事层（与 research/summarize 共用调用封装）
├── output/
│   ├── reports/                # daily_YYYYMMDD.md 存档
│   └── site/                   # 静态站点，每次运行后重新生成
│       ├── index.html          # V1 主仪表盘
│       ├── rotation.html       # V2
│       ├── stocks.html         # V3
│       ├── alerts.html         # V4
│       └── options.html        # V5 期权分析
├── run_daily.py                # 编排入口：fetch → score → narrate → render
└── scheduler 说明               # cron 配置示例（美东收盘后 30 分钟触发）
```

**关键实现纪律：**

1. yfinance 调用全部走统一封装：指数退避重试、结果缓存当日落库、失败降级
   （某维度取数失败时该维度标记"数据缺失"并从当日综合分中剔除权重，绝不静默用旧数据冒充）
2. Claude API 调用集中在 narrative 层一个文件里，方便控制成本和替换模型
3. HTML 用模板 + 数据注入的方式生成（Jinja2），不要在 Python 里拼 HTML 字符串
4. 静态站点可扔 GitHub Pages / Vercel，无后端服务

## 7. 前端设计规范

> 本节基于 design-taste-frontend skill 适配。注意：该 skill 自身声明 dashboard 类
> 界面不在其主要适用范围（它面向 landing page / portfolio），因此本节采用
> "架构遵循其设计系统选择原则 + 视觉纪律移植其硬规则" 的方式，而非全文照搬。

### 7.1 Design Read（按 skill 第 0 节要求先声明）

**Reading this as: 个人量化决策数据面板 for 单一专业用户（使用者本人），
with a 冷静克制的 dark-tech 数据语言，leaning toward Carbon 式数据密度纪律 +
自建 token 的轻量实现。**

三个 Dial 取值（按 skill 1.A 推断表，本项目属 trust-first + 数据密集场景）：

- **DESIGN_VARIANCE: 3** —— 数据面板要的是稳定可预期的网格，不是艺术排版
- **MOTION_INTENSITY: 2** —— 仅 hover/状态切换反馈，无滚动动画、无循环动效
- **VISUAL_DENSITY: 6** —— 偏 cockpit 密度，等宽数字字体，紧凑但留有呼吸

### 7.2 设计系统选择

按 skill 2.A 的 honesty rule：本项目是数据密集面板，最匹配的官方选择是
**@carbon/react（IBM Carbon）**——成熟的数据密度模式和内建可访问性。

但考虑到本项目输出是**无后端的静态 HTML**（不跑 React 构建链），采用降级方案：
**原生 CSS 自建 token 系统 + Chart.js 图表**，视觉语言向 Carbon 的数据面板
纪律看齐（紧凑行高、等宽数字、克制的语义色）。若后续迁移到 React/Next.js
交互版，则直接上 @carbon/react，不要自己重造组件。

### 7.3 Token 系统（全站统一，写进一个 CSS 文件）

**色彩（遵循 skill 4.2 色彩校准）：**

- 背景基调：off-black（#0a0c10 系，禁纯黑 #000000）
- 面板：#12151b，边框 #1f242e
- 文字三级：#e8eaed / #8b93a1 / #565d6b
- **单一 accent**：#5b8def（冷蓝，饱和度 <80%，遵循 LILA rule 不用 AI 紫）
- 语义色仅用于数据含义，不用于装饰：risk-on 绿 #3ddc84 /
  risk-off 红 #ff5c5c / 中性黄 #f0b429
- Color Consistency Lock：全部四个页面（V1-V4）共用同一套 token，
  禁止某个页面私自换 accent

**排版（遵循 skill 4.1，含 Inter 例外说明）：**

- 数字/代码/时间戳：IBM Plex Mono（等宽，数据面板的核心可读性来源）
- 正文/标签：本项目属 skill 4.1 中"neutral/数据工具"例外场景，
  Inter 可用；若想更有辨识度可换 Geist
- **禁 serif**：本项目无 editorial/luxury 属性，任何位置不出现衬线字体
- 行高纪律：数据行 1.4-1.5，叙事段落 1.7

**形状：**

- Shape Consistency Lock：全站统一 8-12px 一档圆角，卡片/按钮/标签同一体系
- 阴影：不用。暗色面板靠 1px 边框和背景层次区分，不做投影

### 7.4 布局纪律

- 最大宽度 1180-1400px 居中容器
- 信息层级固定三层：结论层（综合分+状态）→ 分解层（维度卡片）→ 明细层（图表/表格）
  每个页面都遵循这个"先给答案再给依据"的阅读顺序
- 长列表（>5 项）按 skill 4.9：用表格组件或分组卡片，不用裸 `<ul>` 无限拉长
- 每个数据卡片必须完整实现四态（skill 4.5）：loading（骨架屏，非 spinner）、
  empty（"该维度今日数据缺失"，指明原因）、error（可读的错误+重试指引）、正常态

### 7.5 AI Tells 禁令（skill 第 9 节移植，数据面板语境）

以下在本项目全部页面中禁止出现：

- ❌ em-dash / en-dash（skill 9.G 零容忍规则；中文语境用中文标点，
  范围用连字符如 "35-65"）
- ❌ 纯黑 #000000、纯白 #ffffff（用 off-black / off-white）
- ❌ neon 外发光、渐变文字、AI 紫蓝渐变背景
- ❌ 装饰性 status 圆点（语义色圆点仅在真正表达状态时使用，图例除外）
- ❌ 假精确数字（99.99% 式的编造数据；原型阶段的示例数据必须显式打
  "示例数据 · 非实时" 标签，正式版删除）
- ❌ 每行都 border-t + border-b 的列表样式
- ❌ 无限循环动画（数据面板不需要任何 loop 动效）
- ❌ 手绘 SVG 图标（需要图标时用 @phosphor-icons 或 @tabler/icons 的 CDN 版本，
  全站一个家族，统一 strokeWidth）
- ❌ 版本标签（V0.6 / BETA）出现在页面主视觉区

### 7.6 可访问性与暗色模式（skill 第 6/8 节）

- 本项目默认暗色。若做双模式，用 CSS variables 一套 token 切换，
  两种模式都要过 WCAG AA 对比度（正文 4.5:1）
- prefers-reduced-motion：MOTION_INTENSITY 已定 2，仅剩的过渡动效也要
  在该媒体查询下收敛为直接切换
- 图表色板必须同时用形状/位置冗余编码（红绿色盲场景下，
  涨跌不能只靠红绿区分，配合 +/- 符号和位置）

## 9. 风险与已知限制（写给未来的自己）

1. **yfinance 是非官方库**，Yahoo 接口变动会导致断数据；fetcher 层的降级标记
   机制是为此准备的，接口挂掉时报告要明确显示"某维度缺失"而不是静默出错误数据
2. **这套评分体系没有经过历史回测验证**，M2 校准期的人工对照就是最初的验证方式；
   不要在校准完成前把它当成决策的唯一依据
3. **舆情模块的 LLM 归纳存在误读风险**，溯源标题机制是为了让使用者能一键核对原文
4. **A股二期接入时不要复用美股的阈值**，两个市场的波动特征、涨跌停制度、
   情绪指标体系都不同，config 按 market 分节
